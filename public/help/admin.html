Admin Configuration Guide

NATS connections, relay settings, and deployment examples

Accessing Plugin Settings

Navigate to System Console > Plugins > Cross Guard. The plugin must be enabled before configuration is accessible. Settings are organized into two sections: NATS Configuration and Relay Settings.

NATS Configuration

Understanding Inbound vs Outbound

DirectionFlowUse When
Outbound Mattermost → NATS Your server is the sender. Publishes messages to NATS.
Inbound NATS → Mattermost Your server is the receiver. Subscribes and receives messages from NATS.

A server can have both inbound and outbound connections simultaneously for bidirectional relay. Multiple connections of the same direction are supported for multi-server topologies.

Validation Rules Summary

  1. Names must be globally unique across inbound AND outbound connections
  2. Names must be lowercase alphanumeric with hyphens (no spaces, underscores, uppercase)
  3. Subject must always start with crossguard.
  4. Both sides of a relay must use the same NATS subject
  5. Auth credentials must be complete (both username AND password for credentials auth)
  6. TLS cert and key must be provided as a pair

Relay Settings

Username Lookup

SettingUsernameLookup
TypeBoolean
Defaulttrue

When enabled: Inbound messages look for a local user with the same username as the remote sender. If found, the message is posted as that local user. This creates a seamless experience where messages appear to come from the correct person.

When disabled: All inbound messages are posted by the Cross Guard relay bot, with the remote username shown in the message formatting. Use this if usernames don't match across servers.

Restrict to System Admins

SettingRestrictToSystemAdmins
TypeBoolean
Defaultfalse

When enabled: Only System Admins can run init-team, teardown-team, init-channel, teardown-channel, reset-prompt, reset-channel-prompt, and rewrite-team. Team Admins and Channel Admins are denied access.

When disabled: Team Admins can manage team connections, Channel Admins can manage channel connections (the default permission hierarchy).

Tip

Enable this setting in high-security environments where only centralized admins should control cross-domain relay.

File Transfer Settings

File transfer is configured per connection, not globally. Each NATS connection can independently enable or disable file attachment relay and apply extension filters.

Requirements

  • JetStream-enabled NATS server: File transfer uses NATS JetStream Object Store. Core NATS alone is not sufficient. The NATS server must have JetStream enabled.
  • Object Store bucket: The plugin auto-creates a crossguard-files bucket with a 1-hour TTL on first use. No manual NATS configuration is needed.
  • Max file size: Governed by the Mattermost server's FileSettings.MaxFileSize configuration (default 100 MB). Files exceeding this limit are silently skipped with a log warning.

Filter Examples

Note

For bidirectional file relay, both the outbound and inbound connections must have file_transfer_enabled set to true. File filters are evaluated independently on each side, so the outbound connection can filter what it sends and the inbound connection can filter what it accepts.

Testing Connections

The admin console UI includes a "Test Connection" button for each configured connection. Tests can be run for outbound (publishes a test message) or inbound (subscribes and confirms).

  • API endpoint: POST /plugins/crossguard/api/v1/test-connection (System Admin only)
  • Outbound test: Publishes a message with a unique ID and confirms flush
  • Inbound test: Subscribes to the subject and confirms the subscription is active
  • Tests use a temporary connection (does not affect the running plugin's connections)

Example Configurations

Dual-Server Setup (Server A sends to Server B)

Server A (sender)

Outbound Connections:
[{
  "name": "relay-to-b",
  "address": "nats://nats.example.com:4222",
  "subject": "crossguard.relay.messages",
  "auth_type": "token",
  "token": "shared-secret-token",
  "file_transfer_enabled": true,
  "file_filter_mode": "allow",
  "file_filter_types": ".pdf,.docx,.png"
}]

Inbound Connections: []

Server B (receiver)

Inbound Connections:
[{
  "name": "relay-from-a",
  "address": "nats://nats.example.com:4222",
  "subject": "crossguard.relay.messages",
  "auth_type": "token",
  "token": "shared-secret-token",
  "file_transfer_enabled": true
}]

Outbound Connections: []

Bidirectional Setup (Both servers send and receive)

Server A

Outbound: [{"name": "to-b", "address": "nats://nats:4222", "subject": "crossguard.a-to-b"}]
Inbound:  [{"name": "from-b", "address": "nats://nats:4222", "subject": "crossguard.b-to-a"}]

Server B

Outbound: [{"name": "to-a", "address": "nats://nats:4222", "subject": "crossguard.b-to-a"}]
Inbound:  [{"name": "from-a", "address": "nats://nats:4222", "subject": "crossguard.a-to-b"}]

Note

Each direction uses a different subject to avoid echo loops. Server A publishes on crossguard.a-to-b and Server B subscribes to it, and vice versa.

Troubleshooting

ProblemSolution
"No NATS connections configured" Add connections in System Console > Plugins > Cross Guard first.
Connection test fails Verify NATS server is reachable, check address/port, verify auth credentials match.
Messages not relaying Run /crossguard status. Verify both team AND channel are initialized. Check NATS subject matches on both sides.
"Connection not found" Connection name is case-sensitive and uses direction:name format (e.g., outbound:relay-to-b).
Orphaned connections If a connection name is changed in config, existing team/channel links reference the old name. Teardown and re-init with the new name.
Inbound messages blocked Check if an inbound prompt was blocked. Run /crossguard reset-prompt <name> to clear it and re-evaluate.
Wrong team receives messages Check team name rewrite settings with /crossguard status. Remote team names must match or be rewritten.
Files not transferring Verify file_transfer_enabled is true on both outbound and inbound connections. Ensure the NATS server has JetStream enabled (not just core NATS).
"File semaphore full" log warnings Too many concurrent file transfers. The limit is 32 simultaneous file operations. Large files or slow NATS connections can cause congestion. Consider reducing file sizes or upgrading NATS bandwidth.
File blocked by filter Check file_filter_mode and file_filter_types on the connection. An allow filter rejects extensions not in the list; a deny filter rejects extensions in the list.
File appears after text message Normal behavior. Files arrive via Object Store watcher asynchronously. The inbound watcher retries post mapping lookup up to 3 times with backoff to wait for the text message to be processed first.
Large files silently dropped Files larger than the server's MaxFileSize setting (default 100 MB) are skipped with a log warning. Check FileSettings.MaxFileSize in the Mattermost server configuration.
