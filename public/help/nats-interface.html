NATS Interface Reference

Wire protocol and message schemas for the Cross Guard NATS bus

Overview

Cross Guard relays messages between Mattermost instances using NATS as the transport layer. Text events (posts, edits, deletes, reactions) are wrapped in a message envelope (serialized as JSON or XML, per connection configuration) and published via core NATS publish/subscribe. File attachments use a separate subsystem: NATS JetStream Object Store, which provides reliable binary object transfer with automatic TTL-based cleanup. Both subsystems share the same NATS connection but operate independently.

Text TransportNATS core publish/subscribe
File TransportNATS JetStream Object Store (bucket: crossguard-files)
Wire FormatJSON or XML (UTF-8 encoded bytes, per connection config) for text; binary for files
Subject Prefixcrossguard. (required for all subjects)
Subject Patterncrossguard.{domain}, e.g. crossguard.high, crossguard.low
Retry PolicyUp to 3 attempts with exponential backoff (500ms base, 5s max) for text relay
Text ConcurrencyUp to 256 concurrent text relay operations per plugin instance
File ConcurrencyUp to 32 concurrent file upload/download operations per plugin instance

Message Envelope

Every message published to NATS uses this standard envelope. The type field identifies the message kind. Exactly one payload field is present per message. The envelope can be serialized as JSON or XML depending on the outbound connection's message_format setting. Inbound messages are auto-detected by inspecting the first non-whitespace byte.

NATS MessageEnvelope

Schema

typestring, required. One of the message type values. Determines which payload field is present.
timestampstring, required. UTC timestamp in RFC 3339 format (e.g. 2026-03-27T14:30:00Z). Set when the envelope is created.
post_messageobject, optional. Present when type is crossguard_post or crossguard_update. See PostMessage.
delete_messageobject, optional. Present when type is crossguard_delete. See DeleteMessage.
reaction_messageobject, optional. Present when type is crossguard_reaction_add or crossguard_reaction_remove. See ReactionMessage.
test_messageobject, optional. Present when type is crossguard_test. See TestMessage.

{
  "type": "crossguard_post",
  "timestamp": "2026-03-27T14:30:00Z",
  "post_message": {
    "post_id": "8swgtrrdiff89yj1z33pofo15r",

Message Types

The type field in the envelope acts as a discriminator. Each value maps to a specific payload field in the envelope.

Type ValueGo ConstantPayload SchemaDescription
crossguard_post MessageTypePost PostMessage A new post was created
crossguard_update MessageTypeUpdate PostMessage An existing post was edited
crossguard_delete MessageTypeDelete DeleteMessage A post was deleted
crossguard_reaction_add MessageTypeReactionAdd ReactionMessage A reaction was added to a post
crossguard_reaction_remove MessageTypeReactionRemove ReactionMessage A reaction was removed from a post
crossguard_test MessageTypeTest TestMessage Connectivity test message

Connection Configuration

Each NATS connection (inbound or outbound) is configured in the System Console. This schema is stored in plugin settings and is not transmitted over the wire. It is included here for reference when setting up integrations.

CONFIG NATSConnection

Fields

FieldTypeRequiredDescription
name string Yes Connection identifier. Must start and end with a lowercase letter or number. Hyphens allowed between segments, no consecutive hyphens (pattern: ^[a-z0-9]+(-[a-z0-9]+)*$).
address string Yes NATS server address (e.g. nats://nats:4222).
subject string Yes NATS subject. Must start with crossguard. (e.g. crossguard.high).
tls_enabled boolean No Enable TLS encryption. Default: false.
auth_type string No Authentication method: none, token, or credentials. Default: none.
token string Conditional Bearer token. Required when auth_type is token.
username string Conditional Username. Required when auth_type is credentials.
password string Conditional Password. Required when auth_type is credentials.
client_cert string No Filesystem path to the client TLS certificate.
client_key string No Filesystem path to the client TLS private key.
ca_cert string No Filesystem path to the CA certificate for server verification.
file_transfer_enabled boolean No Enable file attachment relay via JetStream Object Store. Default: false.
file_filter_mode string No File extension filter: "" (no filter), "allow" (whitelist), or "deny" (blacklist). Default: "".
file_filter_types string Conditional Comma-separated extensions (e.g. .pdf,.docx,.png). Case-insensitive. Required when file_filter_mode is "allow" or "deny".
message_format string No Wire format for outbound messages: "json" (default) or "xml". Use XML when sending through a Cross Domain Solution (CDS). Inbound messages are auto-detected regardless of this setting.

File Transfer Protocol

File attachment relay uses NATS JetStream Object Store as an out-of-band binary transfer mechanism. Text and file relay are completely independent systems that share the same NATS connection but use different protocols. File operations cannot starve text relay because they use a separate semaphore.

Object Store Configuration

ParameterValueDescription
Bucket Name crossguard-files Auto-created on first use via CreateOrUpdateObjectStore
TTL 1 hour Objects are automatically purged after TTL. No receiver-side deletion (supports fan-out to multiple receivers).
Object Key {postID}/{randomID} Post ID from the originating server combined with a generated unique identifier. Avoids filename encoding issues.
Max File Size Server's MaxFileSize Governed by FileSettings.MaxFileSize in the Mattermost server config. Default: 100 MB.

Object Metadata Headers

Each uploaded object carries three custom NATS headers that the receiving instance uses to route and attach the file to the correct local post.

HeaderTypeDescription
X-Post-Id string The originating server's post ID. Used to look up the local post via the post ID mapping (pm-{connName}-{remotePostID}).
X-Conn-Name string The connection name that uploaded the file. Receivers filter objects by this header to process only files from the matching inbound connection.
X-Filename string Original filename of the attachment as it appeared on the sending server.

File Filtering

Each connection can independently filter files by extension using three modes:

ModeBehaviorExample
"" (empty, default) No filtering. All files pass. "file_filter_mode": ""
"allow" Only files with listed extensions are relayed. "file_filter_mode": "allow", "file_filter_types": ".pdf,.docx"
"deny" Files with listed extensions are blocked. "file_filter_mode": "deny", "file_filter_types": ".exe,.bat"

Extensions are case-insensitive and a leading dot is optional (both pdf and .pdf are accepted). Filters are evaluated independently on outbound (before upload) and inbound (before download).

JetStream Requirement

File transfer requires a JetStream-enabled NATS server. If the NATS server does not have JetStream enabled, file transfer will fail silently with errors logged. Core NATS text relay is not affected.

XML Format

When message_format is set to "xml" on an outbound connection, messages are serialized using the XML schema defined in schema/crossguard.xsd (namespace: urn:mattermost-crossguard). The root element is CrossguardMessage with child elements matching the envelope fields. An XSAT report is available at schema/crossguard-xsat-report.md for CDS approval processes.

Key Differences from JSON

AspectJSONXML
Root element N/A (top-level object) <CrossguardMessage xmlns="urn:mattermost-crossguard">
Field names snake_case (post_message, channel_name) PascalCase (PostMessage, ChannelName)
Message body field message MessageText
Character escaping JSON string escaping XML entity escaping (&lt;, &gt;, &amp;)
Size overhead Baseline ~30-50% larger (closing tags, namespace)

Inbound Auto-Detection

Inbound connections do not require format configuration. The plugin inspects the first non-whitespace byte of each incoming message: if it is <, the message is parsed as XML; otherwise, it is parsed as JSON. This allows a single inbound connection to accept messages from sources using either format.

BOM Handling

XML output does not include a Byte Order Mark (BOM). If a CDS device prepends a UTF-8 BOM (0xEF 0xBB 0xBF) to the message, the inbound parser will strip it automatically before format detection. BOM-prefixed XML is handled correctly.

Loop Prevention

Cross Guard prevents infinite relay loops through multiple mechanisms:

  • Relayed property: Every post created by inbound message processing is tagged with a crossguard_relayed post property. Outbound hooks skip posts with this property.
  • System message filtering: System-generated messages (join/leave, header changes, etc.) are never relayed.
  • Bot post filtering: Posts from the relay bot itself are excluded from outbound processing.
  • Connection linkage validation: Messages are only published to outbound connections that are explicitly linked to the originating team. Inbound messages are only processed if the target team and channel are linked to the receiving inbound connection.
